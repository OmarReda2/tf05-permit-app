Context:
The MVP functionality is now complete and deployed. The next goal is to significantly improve the visual identity, usability, smoothness, responsiveness, and perceived quality of the app without changing business logic.

The app is a construction safety Permit-to-Work system. The UI should feel serious, modern, operational, and premium — like a smart safety control room for site permits.

Important senior constraint:
Do not rewrite the app.
Do not change Firestore models.
Do not change business logic.
Do not change authentication or role behavior.
Do not break existing functionality.
Do not add unnecessary dependencies.
Use the existing Angular + SCSS structure.

Goal:
Upgrade the app UI/UX across desktop and mobile so it feels polished, cohesive, and domain-specific.

Design direction:
- Professional construction safety / HSE identity
- Modern control-room feeling
- Clear hierarchy
- Strong status visibility
- Smooth interactions
- Clean cards and panels
- Better spacing and typography
- Better mobile experience
- Better empty/loading/error states
- Better forms and tables/cards
- Less generic placeholder feel

Visual identity:
- Add a consistent design system using SCSS variables or CSS custom properties.
- Use a serious palette suitable for safety/construction:
  - dark navy / charcoal base
  - safety orange or amber accents
  - green for approved/active
  - red for rejected/expired
  - blue for pending/info
- Improve typography using system fonts only unless already available.
- Improve spacing, shadows, borders, radius, and visual grouping.
- Add subtle background depth, but keep readability high.

Global UI improvements:
- Improve app shell/sidebar/header/mobile nav.
- Add better active navigation state.
- Add stronger app identity/logo text area.
- Add role/user display styling.
- Improve page headers and section titles.
- Standardize buttons, cards, badges, form fields, tables, and empty states.
- Add consistent loading and error presentation.
- Add subtle CSS transitions/microinteractions.
- Keep animations lightweight and professional.

Screen improvements:
1. Login
- Make it feel like a secure safety permit portal.
- Improve layout, identity, spacing, and error state.
- Keep it simple and responsive.

2. Dashboard
- Improve summary cards.
- Add clear visual hierarchy.
- Make status counts easier to scan.
- Improve recent permits if present.
- Avoid charts unless already simple and dependency-free.

3. Permit List
- Improve filters layout.
- Improve status and expiry badges.
- Desktop: readable table/list.
- Mobile: strong permit cards.
- Fix spacing and action visibility.

4. Permit Details
- Make it feel like a permit dossier/control record.
- Group details into clear sections.
- Make checklist, approval history, and events easy to read.
- Keep approve/reject/print actions clear.

5. New Permit
- Improve form flow and readability.
- Group sections clearly.
- Make checklist completion feel natural.
- Improve validation messages.
- Keep mobile touch-friendly.

6. Approvals
- Improve approval queue readability.
- Make critical actions visible.
- Make reject reason UX clean.
- Emphasize risk/status/expiry.

7. Admin Screens
- Improve permit types and checklist item management.
- Keep admin screens simple but not ugly.
- Do not overbuild dashboards.

8. Activity Log
- Improve event timeline/list presentation.
- Make actor, role, action, permit number, and time easy to scan.

9. Print
- Do not break print CSS.
- Ensure navigation/actions remain hidden in print.

Responsive requirements:
- Desktop must feel spacious and professional.
- Mobile must not have horizontal scrolling.
- Mobile bottom nav must remain usable.
- Tables should become cards on mobile.
- Buttons must be touch-friendly.
- Approval actions must be easy to tap.

Accessibility/usability:
- Maintain good contrast.
- Do not rely on color only; keep text labels.
- Keep focus states visible.
- Keep forms readable.
- Avoid tiny text and tiny tap targets.
- Avoid excessive animation.

Implementation boundaries:
- Prefer SCSS/CSS improvements and small component/template adjustments.
- Reuse existing components and structure where possible.
- Create shared UI helpers only if they reduce duplication clearly.
- Do not introduce Angular Material, Tailwind, Bootstrap, icon libraries, animation libraries, or chart libraries unless already installed.
- If icons are needed, use simple text/emoji-free CSS or existing assets only.
- Do not change routes.
- Do not change Firestore rules.
- Do not change services unless needed for display formatting only.
- Do not change approval logic.
- Do not change permit creation logic.
